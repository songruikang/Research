# 生成式 UI 技术分析

> 背景：业务部门反馈当前查询结果可视化能力不足，需要提升图表生成的丰富度和质量。
> 本文分析生成式 UI 的主流方案、适用场景，以及对本项目的建议。

## 1. 现状分析

### 当前架构

```
用户自然语言 → NL2SQL → SQL执行 → 查询结果
                                       ↓
                              LLM 生成 Vega-Lite spec
                                       ↓
                              前端 vega-embed 渲染
```

### 已有能力

| 维度 | 现状 |
|------|------|
| 图表类型 | 7 种：bar, grouped_bar, stacked_bar, line, multi_line, area, pie |
| 生成方式 | LLM (OpenAI via Haystack) 生成 Vega-Lite JSON spec |
| 校验 | Vega-Lite v5 JSON Schema 校验 |
| 后处理 | ChartSpecHandler 统一配色、交互、图例、响应式尺寸 |
| 交互 | 图表编辑、调整、Pin 到 Dashboard、拖拽布局 |
| Prompt | 344+ 行图表生成指令，含 7 种图表的详细规则 |

### 痛点（推测）

- 图表类型覆盖不足（缺热力图、散点图、漏斗图、仪表盘、组合图等）
- 生成质量不稳定（LLM 选错图表类型、轴配置不合理）
- 自定义能力弱（业务想微调图表但编辑 UI 有限）
- 无法表达复杂可视化（多图联动、交叉筛选、下钻）

## 2. 三条技术路线

### 2.1 Template UI（模板匹配）

**原理：** 预定义 N 个图表模板，LLM 只做分类——"这个数据用哪个模板 + 填什么字段"。

```
LLM 输出示例:
{
  "template": "time_series_line",
  "x": "report_date",
  "y": "alarm_count",
  "group_by": "alarm_level"
}
```

**优势：**
- 确定性最高——模板经过设计和测试，不会出畸形图表
- Token 消耗最少——LLM 只输出字段映射，几十个 token
- 开发成本低——每个模板是一个写死的 ECharts/Vega 配置函数
- 调试简单——问题只可能出在"选错模板"或"映错字段"

**劣势：**
- 表现力上限 = 模板数量。业务每提一个新需求 = 开发一个新模板
- 无法处理"差一点就行"的场景——模板是离散的，需求是连续的
- 长期维护成本高——模板数量膨胀后难以管理
- 不具备创造性——永远只能展示预设的样式

**适合场景：** 需求高度收敛的标准报表系统。

### 2.2 DSL（领域特定语言）

**原理：** 定义一套中间描述语言，LLM 输出 DSL 文本，渲染引擎解释执行。

DSL 的范围很广，从已有规范到完全自定义：

| DSL 类型 | 代表 | LLM 熟悉度 | 表现力 |
|----------|------|-----------|--------|
| 已有可视化规范 | Vega-Lite, ECharts option | 高（训练数据充足） | 高 |
| 通用生成式 UI DSL | OpenUI Lang | 低（自定义语法） | 通用 UI，不专注可视化 |
| 自建领域 DSL | 自定义 | 零（需要完整 prompt 教学） | 完全可控 |

**优势：**
- 安全——没有任意代码执行，输出空间受控
- Token 效率——比生成完整代码紧凑
- 可校验——有 schema 就能做格式校验
- 可扩展——DSL 语法演进比模板灵活

**劣势：**
- 自定义 DSL 需要设计、实现、维护三重成本
- LLM 对陌生 DSL 的遵循度是核心风险
- 调试链路多一层：用户意图 → LLM → DSL → 渲染

**适合场景：** 需要平衡表现力和可控性的生产系统。

### 2.3 直接生成 Code

**原理：** LLM 直接输出可执行的前端代码（React/HTML/SVG），在沙箱中渲染。

```jsx
// LLM 直接输出:
<ResponsiveContainer width="100%" height={400}>
  <BarChart data={data}>
    <XAxis dataKey="name" />
    <YAxis />
    <Bar dataKey="value" fill="#8884d8" />
  </BarChart>
</ResponsiveContainer>
```

**优势：**
- 表现力无上限——LLM 对 React/HTML/CSS 训练极其充分
- 不需要设计 DSL——直接利用现有组件库
- 灵活性最强——能生成任何可视化效果
- 适合探索性分析——"给我看看这个数据有什么有趣的模式"

**劣势：**
- 安全风险——需要沙箱隔离（iframe sandbox / WebContainer）
- Token 消耗大——完整 React 组件代码几百到上千 token
- 生成质量不稳定——代码语法错误、运行时异常
- 延迟高——生成 + 编译 + 渲染链路长
- 企业环境审批难——"AI 生成的代码直接在浏览器执行"

**适合场景：** 内部原型工具、数据科学 Notebook、对安全要求宽松的创意探索。

### 路线对比总结

| 维度 | Template | DSL | Code Gen |
|------|----------|-----|----------|
| 表现力 | 低（有限模板） | 中-高（取决于 DSL） | 最高（无限制） |
| 确定性 | 最高 | 高 | 低 |
| Token 成本 | 最低 | 中 | 最高 |
| 安全性 | 最高 | 高 | 需要沙箱 |
| LLM 遵循度 | 高（输出简单） | 取决于 DSL 知名度 | 高（代码是强项） |
| 开发成本 | 低启动/高维护 | 中 | 高（沙箱+错误处理） |
| 适合我们 | 短期补丁 | **主路线** | 远期探索 |

## 3. OpenUI 为什么不适合我们

这个问题需要拆成两层回答：**技术层面**和**场景层面**。

### 3.1 抽象层级错配

OpenUI 的核心抽象是 **UI 组件**——Button、Card、Carousel、Form。它解决的问题是"让 LLM 生成一个完整的用户界面"。

我们的核心抽象是 **数据可视化**——Encoding（编码）、Mark（图形标记）、Scale（比例尺）、Axis（轴）。我们解决的问题是"让 LLM 根据数据特征选择合适的图表并正确配置"。

```
OpenUI 的世界观:
  Page → Section → Component(props) → Render

数据可视化的世界观:
  Data → Transform → Encoding(x, y, color, size) → Mark(bar, line, point) → Render
```

这两个世界观的思考方式完全不同。用 OpenUI 做图表，相当于用"组件思维"去包装"编码思维"——你需要把 BarChart 定义为一个组件，把 x/y/color 定义为 props，然后教 LLM 怎么填。但这**恰好就是 Vega-Lite 已经做好的事情**，而且 Vega-Lite 在 LLM 训练数据中大量存在。

### 3.2 LLM 遵循度是硬伤

OpenUI Lang 是 2024-2025 年推出的自定义语法。LLM 的训练数据中几乎没有 OpenUI Lang 的样本。

对比：

| DSL | GitHub 仓库/文档量 | LLM 训练数据覆盖 | 遵循度预期 |
|-----|-------------------|-----------------|-----------|
| Vega-Lite | 大量学术论文 + 教程 + StackOverflow | 充分 | 高 |
| ECharts option | 中文生态最大的图表库，海量示例 | 充分 | 高 |
| OpenUI Lang | 新项目，文档有限 | 几乎为零 | 依赖 prompt 工程 |

OpenUI 通过 `createLibrary` 自动生成 system prompt 来教 LLM 它的语法，但这本质上是**在运行时用 prompt 弥补训练数据的缺失**。对于简单 UI 组件（填几个 props）这够用，但对于数据可视化（需要理解数据类型、选择编码通道、配置比例尺），prompt 教学的上限远低于 LLM 内化的知识。

### 3.3 已有架构的沉没成本

当前项目已经有完整的 Vega-Lite 生成管线：

- 后端：`chart_generation.py` + 344 行 prompt + JSON Schema 校验
- 前端：`ChartSpecHandler` + 配色/交互/图例处理 + Dashboard 集成
- 服务层：异步生成 + 状态轮询 + 缓存 + 调整 API

切换到 OpenUI 意味着重写这整条链路，但**可视化能力并不会因此提升**——因为瓶颈不在渲染框架，而在 LLM 对数据语义的理解。

### 3.4 引入复杂度但不解决核心问题

| 我们的核心问题 | OpenUI 能解决吗 |
|--------------|---------------|
| 图表类型不够丰富 | 不能——需要扩展 Vega-Lite spec 的生成范围 |
| LLM 选错图表类型 | 不能——这是 prompt/few-shot 的问题 |
| 图表配置不合理 | 不能——这是 LLM 对数据理解的问题 |
| 编辑/自定义能力弱 | 部分能——但用 Vega-Lite 的编辑 UI 也能做 |
| 想要更丰富的交互 | 不能——OpenUI 的强项不在数据交互 |

### 3.5 OpenUI 适合的场景（对比说明）

OpenUI 适合的是**通用对话式 UI 生成**场景：
- AI 聊天助手需要动态展示表单、卡片、列表
- 类似 ChatGPT Artifacts 的功能——根据对话生成完整 UI
- 需要在多种 UI 组件间灵活切换

我们的场景更窄更深——只需要"数据 → 最佳图表"，用专业的可视化 DSL（Vega-Lite/ECharts）效率更高。

### 3.6 结论

**OpenUI 不是不好，是不对口。** 它解决的是"通用 UI 生成"问题，我们面对的是"数据可视化"问题。用通用工具解决专业问题，会同时承受通用框架的复杂度和专业场景的不适配。

## 4. 图表生成质量由什么决定

图表质量 = "这张图是否准确、清晰、高效地传达了数据中的信息"。

影响质量的因素可以分为三层：

### 4.1 语义理解层（影响最大）

这一层决定"图表画的对不对"。

**4.1.1 数据语义识别**

LLM 需要理解每一列的语义才能做出合理的可视化决策：

| 列名 | 语义 | 可视化决策 |
|------|------|-----------|
| `report_date` | 时间维度 | → X 轴，temporal 类型 |
| `alarm_count` | 度量值 | → Y 轴，quantitative 类型 |
| `alarm_level` | 分类维度 | → color 编码 |
| `device_id` | 标识符 | → 不适合做轴，适合做筛选 |

如果 LLM 把 `device_id`（几千个取值）映射到 X 轴，图表就废了。

**影响因素：**
- 列名的可读性（`alarm_cnt` vs `c1`）
- 是否提供列的描述/注释（语义层的价值）
- 是否提供 sample data（让 LLM 看到实际取值）
- 是否提供列的 distinct 值数量（帮助判断维度还是度量）

**4.1.2 图表类型选择**

不同数据模式适合不同图表：

| 数据模式 | 适合图表 | 常见错误 |
|----------|---------|---------|
| 时间序列 | 折线图 | 用了柱状图（时间点多时拥挤） |
| 分类比较 | 柱状图 | 用了饼图（分类太多时不可读） |
| 占比分布 | 饼图（<7 类） | 分类 >10 还用饼图 |
| 相关性 | 散点图 | 用了折线图（暗示趋势） |
| 分布 | 直方图/箱线图 | 用了柱状图（丢失分布信息） |
| 多维对比 | 分组柱状图/热力图 | 多条折线图（线太多看不清） |

**影响因素：**
- Prompt 中图表选择规则的清晰度
- Few-shot 示例的覆盖面
- 是否有数据特征的预分析（行数、列数、数据类型分布）

**4.1.3 用户意图理解**

同一份数据，"趋势是什么" vs "各类占比多少" vs "有没有异常" 需要不同的图表。LLM 需要从用户的自然语言问题中提取可视化意图。

### 4.2 Spec 生成层（影响中等）

这一层决定"图表配置是否正确"。

**4.2.1 编码通道配置**

Vega-Lite 的核心是 encoding——把数据字段映射到视觉通道（位置、颜色、大小、形状）。

常见质量问题：
- 数据类型标错（temporal 标成 nominal → 时间轴变分类轴）
- 聚合方式不对（该用 sum 用了 mean，或漏了聚合）
- 排序缺失（柱状图应按值排序而非默认字母序）
- 颜色映射不合理（离散数据用了连续色带，或反过来）

**4.2.2 Transform 正确性**

复杂图表需要数据转换（fold, filter, aggregate, calculate）。LLM 生成 transform 的错误率明显高于简单 encoding。

**4.2.3 Spec 完整性**

缺少 title、轴标签、图例名称、tooltip 会让图表难以理解。LLM 有时生成结构正确但信息不完整的 spec。

**影响因素：**
- Prompt 中的生成规则和约束
- JSON Schema 校验的严格程度
- 后处理逻辑的完善度（当前的 ChartSpecHandler）
- LLM 对 Vega-Lite 语法的掌握程度

### 4.3 渲染呈现层（影响较小但用户感知强）

这一层决定"图表好不好看"。

**4.3.1 样式和配色**

- 色彩方案是否专业（避免刺眼对比色）
- 字体大小和间距是否合理
- 是否有统一的视觉风格

**4.3.2 交互体验**

- 悬停 tooltip 是否信息充分
- 图例点击筛选是否可用
- 大数据量下是否做了截断/分页

**4.3.3 响应式和适配**

- 不同屏幕尺寸下的表现
- 长标签的截断和旋转
- 图表容器的比例

**影响因素：**
- 后处理 handler 的成熟度
- 前端组件的打磨程度
- 这部分和 LLM 关系不大，主要是工程实现

### 4.4 质量提升的优先级

```
投入产出比排序（从高到低）:

1. 增强语义上下文 → 让 LLM 理解数据含义
   - 列描述、业务注释、数据类型、取值范围
   - 这恰好是语义层的价值所在

2. 优化图表选择 Prompt + Few-shot → 让 LLM 选对图表
   - 增加选择规则、反例说明
   - 按数据模式组织 few-shot

3. 增加后处理规则 → 兜底 LLM 的常见错误
   - 饼图 >7 类自动降级为柱状图
   - 时间轴自动识别和格式化
   - 超长类别名截断

4. 扩展图表类型 → 覆盖更多场景
   - 散点图、热力图、漏斗图、组合图
   - 每增加一种需要 prompt + 校验 + 后处理全套

5. 提升渲染层体验 → 好看、好用
   - 配色方案升级、动画、导出
```

## 5. 自建 DSL 的代价与优势

### 5.1 什么是"自建 DSL"

在我们的场景中，自建 DSL 意味着**定义一套比 Vega-Lite 更精简、更贴合业务的图表描述语言**，让 LLM 输出这套语言，然后由我们的渲染引擎翻译成最终图表。

示例（假设的自建 DSL）：

```yaml
chart:
  type: time_series
  title: 近30天告警趋势
  x: report_date
  y: alarm_count
  group: alarm_level
  options:
    show_trend: true
    highlight_anomaly: true
    time_granularity: day
```

对比 Vega-Lite 同样的图表：

```json
{
  "mark": "line",
  "encoding": {
    "x": {"field": "report_date", "type": "temporal", "timeUnit": "yearmonthdate"},
    "y": {"field": "alarm_count", "type": "quantitative", "aggregate": "sum"},
    "color": {"field": "alarm_level", "type": "nominal"}
  },
  "title": "近30天告警趋势"
}
```

### 5.2 代价

#### 5.2.1 设计成本（一次性，高）

- 需要定义 DSL 的语法规范、数据类型系统、图表类型枚举
- 需要处理边界情况：缺省值、冲突配置、无效组合
- 需要迭代——第一版一定不够用，改语法意味着改 prompt + 改渲染器 + 改校验
- 需要文档——DSL 的 spec 文档、示例库、错误码

**估算：** 一个有经验的工程师 2-3 周完成 v1，含基本图表类型。

#### 5.2.2 LLM 教学成本（持续，核心风险）

这是自建 DSL 最大的不确定性。

LLM 对自定义 DSL 的遵循度取决于：

| 因素 | 影响 |
|------|------|
| DSL 是否接近某种 LLM 见过的格式 | YAML/JSON 比自定义语法好 |
| Prompt 中 DSL 规范的长度 | 越长越消耗 context window |
| Few-shot 示例的数量和质量 | 至少需要 10-20 个覆盖主要场景 |
| DSL 的复杂度 | 嵌套越深、规则越多，遵循度越低 |

**对比：** Vega-Lite 不需要 prompt 教学——LLM 已经"会"了。省下的 prompt 空间可以用来放更多业务上下文。

#### 5.2.3 渲染引擎开发成本（一次性，中-高）

- DSL → ECharts/Vega-Lite 的翻译层
- 或 DSL → 直接 SVG/Canvas 渲染（成本更高）
- 每新增一个图表类型 = DSL 语法扩展 + 翻译规则 + 测试

#### 5.2.4 维护成本（持续）

- 业务新需求 → DSL 语法演进 → prompt 更新 → 渲染器更新 → 测试回归
- 模型升级（Qwen 换代、切换供应商）→ 可能需要重新调整 few-shot
- 版本兼容——旧版 DSL 生成的图表在新渲染器下是否正常

### 5.3 优势

#### 5.3.1 Token 效率

自建 DSL 可以比 Vega-Lite 紧凑得多：

| 表达 | Vega-Lite Token | 自建 DSL Token | 节省 |
|------|----------------|---------------|------|
| 简单柱状图 | ~80 | ~30 | 62% |
| 分组折线图 | ~150 | ~50 | 67% |
| 带 transform 的复杂图 | ~300 | ~80 | 73% |

在 NL2SQL 已经消耗一轮 LLM 调用的情况下，图表层的 token 节省有实际的成本意义。

#### 5.3.2 业务语义内置

自建 DSL 可以把业务概念直接编码到语法中：

```yaml
# 自建 DSL 可以直接说
chart:
  type: alarm_trend      # 业务概念，不只是 "line chart"
  highlight_anomaly: true  # 业务能力，不是通用 encoding
  threshold:
    critical: 100
    warning: 50
```

这些在 Vega-Lite 中需要复杂的 transform + layer 组合实现，LLM 容易出错。自建 DSL 把复杂度从 LLM 侧转移到渲染引擎侧。

#### 5.3.3 错误空间更小

Vega-Lite spec 的合法空间非常大（几十种 mark、几十个 encoding 属性、嵌套的 transform）。LLM 输出越复杂的 spec，出错概率越高。

自建 DSL 可以通过收窄语法来减少 LLM 犯错的可能：
- 枚举所有允许的图表类型（不允许自由组合）
- 限制 encoding 通道（最多 x, y, color, size）
- 预设合理默认值（不要求 LLM 指定每个属性）

#### 5.3.4 渲染质量可控

DSL → 渲染的翻译层由我们编写，意味着：
- 每种图表类型都是精心设计和测试过的
- 可以统一处理配色、响应式、交互、动画
- 可以加入业务规则（如告警级别的颜色映射是固定的）

这比依赖 LLM 直接生成完美的 Vega-Lite spec 更可靠。

#### 5.3.5 扩展到非图表可视化

自建 DSL 可以超越传统图表，加入：
- 指标卡（KPI card）
- 表格（带条件格式）
- 拓扑图（网络设备关系）
- 组合布局（上面指标卡 + 下面趋势图）

这些在 Vega-Lite 中要么做不到，要么极其复杂。

### 5.4 自建 DSL vs 继续用 Vega-Lite 的决策矩阵

| 判断维度 | 选 Vega-Lite | 选自建 DSL |
|----------|------------|-----------|
| 图表类型需求 | < 15 种标准图表 | 需要业务专属图表或组合布局 |
| LLM 模型 | GPT-4/Claude（对 Vega-Lite 训练充分） | Qwen 等对 Vega-Lite 训练较少的模型 |
| 团队前端能力 | 弱（不想写渲染引擎） | 有前端支持或愿意投入 |
| Token 预算 | 宽松 | 紧张（尤其是自部署模型） |
| 迭代速度要求 | 快——先跑起来 | 稳——愿意投入基建换长期收益 |
| 可视化的核心竞争力 | 不是——SQL 准确率才是 | 是——可视化是差异化卖点 |

## 6. 方案推荐与路线图

### 6.1 推荐路线：渐进式演进

不需要二选一。从当前 Vega-Lite 架构出发，逐步向自建 DSL 演进：

```
Phase 0（当前）     Phase 1              Phase 2              Phase 3
Vega-Lite 直出  →  Vega-Lite + 后处理增强  →  轻量 DSL 封装层  →  完整 DSL + 渲染引擎
                  (快速见效)              (渐进过渡)          (长期目标)
```

### Phase 1：增强现有 Vega-Lite 管线（推荐先做）

**目标：** 不改架构，快速提升图表质量。

- 扩展图表类型到 12-15 种（散点图、热力图、漏斗图、指标卡、组合图）
- 增强 prompt 中的图表选择规则 + 反例
- 增加 few-shot 示例（按数据模式分组）
- 强化后处理：自动降级（饼图 >7 类 → 柱状图）、智能默认值
- 传入更丰富的数据上下文（列描述、取值范围、distinct count）

### Phase 2：薄 DSL 封装层

**目标：** 在 LLM 和 Vega-Lite 之间加一层业务 DSL，降低 LLM 的输出复杂度。

```
LLM → 业务 DSL (JSON, 精简) → 翻译层 → Vega-Lite spec → vega-embed 渲染
```

- DSL 用 JSON 格式（LLM 最熟悉）
- DSL 只暴露业务语义（chart type + field mapping + options）
- 翻译层负责展开为完整的 Vega-Lite spec
- 保留 Vega-Lite 作为渲染层——不重复造轮子

### Phase 3：完整 DSL + 多渲染后端

**目标：** 如果 Phase 2 验证了 DSL 路线可行，扩展到完整能力。

- 支持组合布局（多图联动、dashboard 模板）
- 支持多渲染后端（ECharts 做复杂交互，Vega-Lite 做声明式图表）
- 支持非图表可视化（拓扑图、KPI 卡片、表格）

### 6.2 不建议做的事

- 现阶段引入 OpenUI——解决的不是我们的问题
- 直接跳到 Phase 3——没有 Phase 1/2 的数据积累，DSL 设计会脱离实际需求
- 直接生成代码——安全和稳定性代价太高，且不解决"选对图表"这个核心问题
