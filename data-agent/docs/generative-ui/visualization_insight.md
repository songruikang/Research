# 电信网管可视化洞察：从数据展示到决策支持

> 当前 chart-engine 做的是"数据格式转换"（行列 → 图表），信息密度没有增加。
> 本文分析电信网管场景下可视化真正该做什么，以及行业标杆的做法。

## 1. 当前问题：图表 ≠ 洞察

### 当前做了什么

```
SQL 结果 → Profiler → Selector → ECharts option
```

输出：柱状图、折线图、饼图、散点图、KPI 卡片。

### 为什么不够

| 场景 | 当前输出 | 用户真正想看的 |
|------|---------|--------------|
| "各厂商设备数" | 柱状图（CISCO 9, HUAWEI 6...） | 看表格就行，柱状图没有增加信息 |
| "CPU 利用率趋势" | 折线图 | 哪些时间点超过阈值？跟哪些业务事件关联？ |
| "SLA 达标率" | 数字（92%） | 哪些 VPN 不达标？根因是时延还是丢包？影响哪些客户？ |
| "隧道时延分布" | 散点图 | 时延高的隧道经过了哪些节点？跟链路负载有没有关系？ |

**核心差距：当前可视化只回答"是什么"，不回答"为什么"和"怎么办"。**

## 2. 行业标杆做法

### 2.1 Cisco Nexus Dashboard / Crosswork

**核心理念：拓扑即入口**

- 首屏是网络拓扑图（站点 → 设备 → 接口），不是图表
- 拓扑图上的每个节点带实时状态色（绿/黄/红）
- 点击设备 → 展开该设备的 KPI 面板（CPU/内存/接口利用率）
- 点击链路 → 展开链路的时延/丢包/带宽利用率
- 告警直接标注在拓扑上，一眼看到"哪里出了问题"

**关键设计模式：**

```
全局拓扑（鸟瞰） → 区域聚焦 → 单设备详情 → KPI 趋势
     ↓ 异常高亮         ↓ 关联分析        ↓ 根因定位
```

### 2.2 Splunk Infrastructure Monitoring

**核心理念：指标 × 上下文 = 洞察**

- **流式架构**：实时指标不是 T+1 报表
- **300+ 预建仪表盘**：开箱即用，不需要用户自己配
- **全栈关联**：基础设施指标 ↔ 应用日志 ↔ 业务指标，跨层钻取
- **动态阈值**：不是静态告警线，而是基于历史基线的异常检测
- **Service Map**：自动发现服务依赖关系，从服务视角（而非设备视角）看健康状态
- **零上下文切换**：从告警 → 关联指标 → 日志 → 根因，不需要跳转到不同系统

**关键设计模式：**

```
Overview Dashboard → 告警/异常聚焦 → 上下文展开 → 根因定位
  全局健康状态         智能排序          关联指标+日志     建议动作
```

### 2.3 Grafana Node Graph

**核心理念：关系可视化**

- 节点 + 边的图结构，天然适合网络拓扑
- 节点可以带统计值（CPU/内存/流量）、颜色编码（健康状态）、图标
- 边可以带流量/时延/丢包信息
- 支持点击展开、钻取到详细面板
- 数据格式明确：nodes 表 + edges 表

## 3. 电信网管数据的拓扑关系

我们的 14 张表天然包含三层拓扑：

### 3.1 物理拓扑

```
t_site（站点）
  └── t_network_element（设备）
        ├── t_board（板卡）
        └── t_interface（接口）
              └── t_physical_link（物理链路，连接两个接口）
```

**可视化**：站点地图 → 站点内设备列表 → 设备间物理连接

### 3.2 逻辑拓扑

```
t_l3vpn_service（VPN 业务）
  └── t_vpn_pe_binding（VPN-PE 绑定）
        └── t_network_element（PE 设备）
              └── t_tunnel（隧道，连接两个设备）
                    └── t_srv6_policy（SRv6 策略）
```

**可视化**：VPN 业务 → 绑定的 PE 设备 → 设备间隧道路径

### 3.3 性能度量层

```
t_ne_perf_kpi       → 挂在 t_network_element 上
t_interface_perf_kpi → 挂在 t_interface 上
t_tunnel_perf_kpi   → 挂在 t_tunnel 上
t_vpn_sla_kpi       → 挂在 t_l3vpn_service + t_network_element 上
```

**可视化**：KPI 数据作为拓扑节点/边的"装饰"——颜色、数值标签、趋势小图

## 4. 可视化应该分三个层次

### Level 1：状态总览（Dashboard）

**用途**：运维人员一打开就知道"系统整体健不健康"

**内容**：
- 关键 KPI 卡片：在线设备数 / 告警数 / SLA 达标率 / 平均 CPU
- 健康度热力图：每个站点一个色块，颜色 = 该站点设备的综合健康度
- 异常列表：最近 N 个异常事件，按严重程度排序
- 趋势缩略图：全网 CPU/带宽/时延的 24h 趋势线

**交互**：点击任何指标 → 钻取到 Level 2

### Level 2：拓扑 + 关联（核心创新点）

**用途**：定位问题发生在哪里，关联了什么

**内容**：
- 网络拓扑图（站点 → 设备 → 链路）
- 节点颜色 = 健康状态（CPU/内存综合评分）
- 边粗细 = 带宽利用率，颜色 = 时延等级
- 异常节点/边闪烁或红色高亮
- 点击设备 → 侧边面板展开：KPI 趋势 + 承载的 VPN 列表 + 关联告警
- 点击链路 → 侧边面板展开：带宽/时延/丢包趋势 + 经过此链路的隧道

**关键能力**：
- "影响分析"：点击一个故障设备 → 高亮所有受影响的 VPN 业务和客户
- "路径追踪"：选择一个 VPN 业务 → 高亮其经过的所有 PE 设备和隧道
- "对比视图"：选两个时间点 → 拓扑图上显示指标变化（变好/变差）

### Level 3：根因分析（AI 驱动）

**用途**：回答"为什么出了问题"和"建议怎么处理"

**内容**：
- 异常检测：基于历史基线自动标记异常指标
- 关联推理：时延上升 → 关联同时段的 CPU 上升 / 链路拥塞 / 板卡故障
- 根因排序：最可能的根因排在最前面，附带置信度
- 建议动作：如"建议扩容 PE-BJ-02 到 PE-SH-01 之间的链路"

**这一层需要 LLM 参与。**

## 5. 技术实现路径

### 5.1 拓扑图技术选型

| 方案 | 优势 | 劣势 | 适合 |
|------|------|------|------|
| ECharts Graph | 已集成 ECharts，零新依赖 | 大规模拓扑性能差，交互有限 | 小拓扑（< 100 节点）|
| AntV G6 | 阿里图可视化引擎，性能好，中文生态 | 新依赖，学习成本 | 中等拓扑 |
| React Flow | WrenAI 已在用（数据血缘图） | 偏向流程图，网络拓扑需要定制 | 可复用现有代码 |
| D3.js force layout | 最灵活，社区大 | 开发量大，需手写大量交互 | 完全自定义 |
| Grafana Node Graph 嵌入 | 成熟的拓扑方案 | 需要嵌入 Grafana，架构变复杂 | 如果已有 Grafana |

**推荐：ECharts Graph（先用）→ AntV G6（后续升级）**

理由：
- ECharts 已集成，graph 类型开箱可用
- 14 表的拓扑规模不大（25 站点 + 52 设备 + 100 链路），ECharts 够用
- 后续如果要做大规模拓扑（100+ 表的生产环境），再升级到 G6

### 5.2 数据流

```
用户提问（如"哪些 VPN 业务 SLA 不达标"）
    │
    ▼
NL2SQL → SQL 执行 → 查询结果
    │
    ▼
chart-engine 增强：
    ├── 传统图表（柱/线/饼）→ 当前已有
    ├── 拓扑图 → 新增：从查询结果 + 关联表构建 nodes/edges
    └── 组合视图 → 新增：拓扑 + 指标面板组合
    │
    ▼
前端渲染：
    ├── ECharts（图表 + 简单拓扑）
    └── 侧边详情面板（点击节点/边展开）
```

### 5.3 chart-engine 需要的演进

| 当前 | 下一步 |
|------|--------|
| 输入：单个 SQL 结果 | 输入：主查询结果 + 关联上下文（设备信息、站点信息） |
| 输出：单个 ECharts option | 输出：组合视图（拓扑图 + KPI 面板 + 趋势图） |
| Selector：按数据特征选图表类型 | Selector：识别"这是拓扑类查询"并选择拓扑视图 |
| Builder：生成传统图表 | Builder：新增拓扑图构建（nodes/edges 从关联表获取） |

### 5.4 拓扑图的数据结构

```json
{
  "type": "topology",
  "nodes": [
    {"id": "NE-001", "name": "PE-BJ-01", "category": "PE", "site": "北京核心",
     "status": "UP", "cpu": 45.2, "memory": 62.1},
    {"id": "NE-002", "name": "PE-SH-01", "category": "PE", "site": "上海核心",
     "status": "UP", "cpu": 78.5, "memory": 55.3}
  ],
  "edges": [
    {"source": "NE-001", "target": "NE-002", "type": "physical_link",
     "bandwidth_mbps": 10000, "utilization_pct": 72.5, "latency_ms": 12.3}
  ],
  "highlights": ["NE-002"],
  "kpi_panels": [
    {"node_id": "NE-002", "title": "PE-SH-01 CPU 趋势",
     "chart_type": "line", "data": [...]}
  ]
}
```

## 6. 分阶段路线图

### Phase 1：拓扑基础能力（建议先做）

- chart-engine 新增 `topology` 图表类型
- Selector 识别拓扑类查询（涉及 t_physical_link / t_tunnel / t_vpn_pe_binding）
- Builder 从查询结果构建 nodes/edges
- 前端用 ECharts graph 渲染
- 支持点击节点展开侧边 KPI 面板

**交付物**：在 Chart SQL 页面中，跑"查看所有 PE 设备之间的物理链路"能看到拓扑图

### Phase 2：状态着色 + 指标关联

- 拓扑图节点/边根据实时 KPI 着色
- 点击节点 → 侧边面板展开该设备的性能趋势
- 影响分析：选中故障设备 → 高亮受影响的 VPN 业务

**交付物**：拓扑图上一眼看到"哪里有问题"，点击就能看到详情

### Phase 3：Dashboard 总览

- 预建仪表盘（类似 Splunk 的开箱即用）
- KPI 卡片 + 健康热力图 + 异常列表 + 趋势缩略图
- 点击钻取到拓扑视图

**交付物**：运维人员首屏总览

### Phase 4：LLM 驱动的根因分析

- 异常检测（基线对比）
- 跨指标关联推理
- 自然语言解释 + 建议动作

**交付物**："PE-BJ-02 CPU 利用率异常升高，原因可能是接口 GE0/0/1 的入方向流量激增（带宽利用率 92%），建议检查该接口连接的客户 VPN 流量"

## 7. 与 Cisco/Splunk 的对标

| 能力 | Cisco Nexus | Splunk | 我们当前 | Phase 1-2 后 |
|------|------------|--------|---------|-------------|
| 传统图表 | ✓ | ✓ | ✓ | ✓ |
| 网络拓扑图 | ✓（核心能力） | ✓（Service Map） | ✗ | ✓ |
| 状态着色 | ✓ | ✓ | ✗ | ✓ |
| 点击钻取 | ✓ | ✓（零切换） | ✗ | ✓ |
| 影响分析 | ✓ | 部分 | ✗ | ✓ |
| 异常检测 | ✓ | ✓（动态阈值） | ✗ | Phase 4 |
| 根因分析 | 部分 | ✓（AI） | ✗ | Phase 4 |
| 自然语言交互 | ✓（Canvas AI） | 部分 | ✓（NL2SQL） | ✓ |

**我们的差异化优势**：NL2SQL + 拓扑可视化。
Cisco/Splunk 的可视化是预定义的仪表盘，用户只能看固定视图。
我们可以做到"用户用自然语言问任意问题 → 系统自动选择最佳可视化方式（图表 or 拓扑 or 组合）"。
