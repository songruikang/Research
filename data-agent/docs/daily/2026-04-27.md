## 今日产出

### 评测环境问题排查与修复

1. **公司云 sqlglot 依赖问题**：venv 中 `pip` 指向系统而非虚拟环境，导致 `sqlglot` 未装入 venv。解决：`python -m pip install sqlglot` 或 `.venv/bin/pip install sqlglot`
2. **时间戳过期导致准确率虚低**：Mac 上的 mock 数据时间基准是 2025-03-29，时间窗口查询（如 `WHERE collect_time >= CURRENT_TIMESTAMP - INTERVAL 7 DAY`）全部返回 0 行。执行 `4_refresh_timestamps.py` 后 unverifiable 从 10% 降至 1-2%
3. **数据生成流程确认**：`1_generate_data.py` → `4_refresh_timestamps.py` → `2_export_init_sql.py` → Docker restart。各环节不会自动同步，需手动依次执行

### 研究课题对齐

经过讨论，明确两个核心研究方向：

**课题一：规模剪裁下的准确率保持（横向扩展）**

- 14 表 ~10K token 全量注入，准确率 82-87%，知识注入无效
- 100+ 表 ~3500 列无法全量塞入 Qwen 32B context
- Schema Linking 和知识注入的真实定位：**不是帮模型理解题目，而是做 context 裁剪**
- 研究问题：什么裁剪策略能做到"裁完等价于全量"——不丢失模型需要的信息

**课题二：复杂场景天花板突破（纵向深入）**

- 当前天花板 87%（Opus 全量+Few-shot+知识），剩余 13% 是模型有完整信息但推理错误
- 错误集中在：二级聚合、多表条件交叉、隐式时间窗口等模式
- 方向：逐模式分析错因，针对性设计方案（分解、自校正、多候选投票），单点突破

两课题关系：课题一决定能否商用（14 表 → 100 表），课题二决定商用后竞争力（87% → 90%+）。

## 今日教训

### 1. 排查问题先理解完整链路

准确率不一致时，我在文件同步、md5、依赖版本上打转，而根因是时间戳未刷新（业务语义层面）。应该先把 `1_generate → 4_refresh → 2_export → docker` 完整链路理清，从"什么会导致同 SQL 不同结果"倒推。

### 2. 记忆必须先验证

用户提到 Schema/Scheme 拼写问题时，未经搜索验证就直接保存为记忆。违反"做对的不做讨好的"原则。可查证的事实应先搜索确认，有分歧应讨论后再记录。

### 3. "语义层"要去魅

行业热词（Semantic Layer、Knowledge Graph for SQL、Context Engineering）本质是元数据管理 + Prompt 工程，属于软件工程范畴。实验数据已证明：对强模型，这些外部信号的边际贡献在噪声范围内。不应把工程活包装成研究课题。
