## 今日教训

### 1. 提交质量差，QA 轮次过多

文件命名不对（generate_sqls 实际是生成 prompt）、AGENTS.md 多次不同步、embedding 硬编码 localhost、Ollama API 路径错误、match_level 没传进 results——每个都是用户指出后才修。

**根因**：写完就交付，没有站在用户视角走一遍完整流程。

**改进**：CLAUDE.md 已加四阶段检查点，提交前必须一次性自检完毕。

### 2. 评测数字反复变化，信任度低

同一组实验的数字改了 3 次（评测标准变了 3 次：严格→子集→列交集），导致用户对"70%+"存疑。最终用 clean rerun 验证。

**根因**：改评测逻辑和跑实验交替进行，前后对比的不是同一把尺子。

**改进**：以后改评测标准后必须 clean rerun 全部实验，不能拿新标准数字和旧标准数字对比。

### 3. 知识注入的反直觉结论

三种知识注入方式全部负优化，但 clean rerun 后 B 组（知识）反而最高（81% 列交集）。LLM 非确定性（±5pp）比 prompt 工程的边际效应更大。

**启示**：对 Opus，prompt 工程到头了，裸模型就是天花板。
